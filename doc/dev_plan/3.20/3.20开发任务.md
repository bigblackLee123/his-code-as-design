# 3.20 开发任务

## 任务一：处方流转 → 治疗终端全链路修复（前后端）

### 问题诊断

医生终端确认处方后，数据链路存在 5 个断裂点，导致治疗终端无法正确接收和执行处方：

| # | 断裂点 | 现状 | 目标 |
|---|--------|------|------|
| 1 | 处方未持久化 | `StatusTransition` 只调 `enqueueTreatment(patientId)`，`prescriptionService.save()` 从未调用 | 流转时同时保存 prescriptions + 拆解 prescription_steps |
| 2 | prescription_steps 从未创建 | DB 表已建好，无代码写入 | 按 selectedProjects 自动拆解，每个项目一条 step（带 region） |
| 3 | 治疗终端拿不到处方项目 | `buildTreatmentPatient()` 用 `therapyService.getProjects().slice(0,3)` 取通用项目 | 按 consultation → prescription → prescription_steps 查该患者的实际处方项目 |
| 4 | 治疗后数据未入库 | PostVitalSigns / PostScaleForm 数据只存前端 state | 写入 `vital_signs(stage=post-treatment)` + `scale_results(stage=post)` |
| 5 | treatment_records 与 steps 未关联 | `callNextTreatment()` 用 `upsert on consultation_id`（1:1），无 region 区分 | 按 region 创建多条 treatment_record，回写 `prescription_steps.treatment_record_id` |
| 6 | 禁忌症未持久化 | `ContraindicationInput` 只存前端 state，`consultation_contraindications` 关联表从未写入 | 流转时写入关联表 |
| 7 | 量表结果未持久化 | `ScaleForm.onSubmit` 只回调到页面 state，`scaleService.saveResult()` 存在但从未调用 | 流转时调用 `scaleService.saveResult()` 写入 DB |

### 额外问题：治疗终端缺少多房间支持

| # | 问题 | 现状 | 目标 |
|---|------|------|------|
| 8 | 无 region 绑定 | 治疗终端无登录/选区概念，不知道自己是哪个房间 | 终端启动时选择 region（睡眠区/情志区/运动疗愈区） |
| 9 | 叫号模式不适用 | 当前是统一叫号，数据流图要求刷卡签到 | 改为按 region 过滤待治疗患者，输入医保卡号后四位模拟刷卡签到（后续接真实读卡器只需换输入源） |
| 10 | 无 Realtime 订阅 prescription_steps | 只订阅了 queue_items | 新增 prescription_steps 表的 Realtime 订阅，房间完成后其他终端实时感知 |
| 11 | 流转检查缺失 | 本房间完成后无后续引导 | 完成后检查全部 steps：未完成 → 提示卡片；全部完成 → 弹出治疗后采集 |

### 刷卡签到模拟方案

由于暂无真实刷卡机，治疗终端的「刷卡签到」采用手动输入模拟：

- 输入方式：医保卡号后四位（如 `0003`）
- 后端查询：`patients.insurance_card_no LIKE '%0003'` + `status IN ('pending-treatment', 'treating')`
- 匹配成功 → 显示患者信息 + 本房间待做项目
- 匹配失败 → 提示「未找到待治疗患者，请确认卡号」
- 后续对接真实读卡器时，只需将输入源从手动改为硬件回调，查询逻辑不变
- 参考实现：分诊终端 `PatientCheckIn.tsx` 的模拟刷卡模式

### 实施步骤

#### 步骤 1：后端 — prescriptionStepsService（新建）

新建 `src/services/supabase/prescriptionStepsService.ts`

```ts
// 核心方法：
createSteps(prescriptionId, projects: TherapyProject[])
  // 遍历 projects，每个写入一条 prescription_steps（region 从 project.region 取）
getStepsByConsultation(consultationId)
  // JOIN prescriptions → prescription_steps → therapy_projects
getStepsByRegion(consultationId, region)
  // 本房间待做项目
startRoomSteps(consultationId, region)
  // 批量更新本房间 pending → in-progress，记录 started_at
completeRoomSteps(consultationId, region, treatmentRecordId)
  // 批量更新本房间 → completed，回写 treatment_record_id
checkAllCompleted(consultationId)
  // 返回 { allDone: boolean, pendingRegions: string[], pendingSteps: [...] }
```

需要新增类型 `PrescriptionStep` 到 `src/services/types.ts`：
```ts
interface PrescriptionStep {
  id: string;
  prescriptionId: string;
  projectId: string;
  projectName: string;    // JOIN 冗余
  region: string;
  sortOrder: number;
  status: 'pending' | 'in-progress' | 'completed' | 'skipped';
  startedAt: string | null;
  completedAt: string | null;
  treatmentRecordId: string | null;
}
```

新增 mapper：`toPrescriptionStep()` / `fromPrescriptionStepInsert()`

#### 步骤 2：后端 — 修改 prescriptionService.save()

1. 保存 prescriptions 记录

#### 步骤 2.5：后端 — 补全医生终端诊疗数据持久化

当前禁忌症和量表结果只存在前端 state，需要在流转时一并持久化：

**禁忌症关联表写入：**
- `contraindicationService` 新增 `saveForConsultation(consultationId, contraindications[])`
- 写入 `consultation_contraindications` 关联表（先 DELETE 旧记录再 INSERT）

**量表结果写入：**
- 在 `StatusTransition.handleConfirm` 中调用已有的 `scaleService.saveResult(patientId, result, 'pre')`
- `scaleService.saveResult()` 方法已实现，只是从未被调用

**流转时的完整持久化顺序：**
```
handleConfirm:
  1. scaleService.saveResult(patientId, scaleResults, 'pre')     // 量表
  2. contraindicationService.saveForConsultation(...)             // 禁忌症
  3. prescriptionService.saveWithSteps(patientId, rx, projects)  // 处方 + steps
  4. queueService.enqueueTreatment(patientId)                    // 入治疗队列
```

这样治疗终端（以及后续的历史记录查询）就能通过 consultationId 聚合拿到完整的诊疗数据。

#### 步骤 2.6：AI 建议链路修复 — 改用 consultationId 模式

**现状问题：**
- Edge Function 后端已完整实现 consultationId 模式：通过 `loadPatientContext()` 从 DB 聚合 vital_signs、consultation_contraindications、scale_results
- Edge Function 已实现 `ai_suggestions` 持久化：保存 model、prompt_snapshot、raw_response、confidence
- 但前端 `AISuggestionPanel` 调用时没传 consultationId，走的是旧模式（前端直传数据）
- 导致 AI 拿到的是前端临时 state 数据，且建议结果未持久化到 `ai_suggestions` 表

**修复方案：**

1. 调整医生终端操作顺序：禁忌症和量表需要在调 AI 之前就持久化
   ```
   医生操作流程（调整后）：
     录入禁忌症 → 即时写入 consultation_contraindications
     提交量表   → 即时调用 scaleService.saveResult()
     点击「获取AI建议」→ 传 consultationId → Edge Function 从 DB 聚合
     AI 返回 → 自动持久化到 ai_suggestions 表
     医生采纳/修改 → 确认处方 → 保存 prescriptions + steps → 入队
   ```

2. 修改 `AISuggestionPanel.fetchSuggestion()`：
   ```ts
   // 改为传 consultationId
   const consultationId = await consultationHelper.getActiveId(patient.id);
   const result = await aiService.getTherapySuggestion({ consultationId });
   ```

3. 修改 `ContraindicationInput`：选择/移除禁忌症时即时写入 DB（而非等到流转时）
4. 修改 `ScaleForm`：提交量表时即时调用 `scaleService.saveResult()`

**这样 AI 就能：**
- 从 DB 拿到完整的患者历史信息（生理数据、禁忌症、量表评分）
- 建议结果（prompt_snapshot + raw_response + confidence）全部记录在 `ai_suggestions` 表
- 后续可通过 `ai_suggestions` 表分析 AI 推荐质量，做持续改进

#### 步骤 3：后端 — 修改 treatmentQueueService

**enqueueTreatment(patientId, projects)**：
- 新增 projects 参数
- 先调 `prescriptionService.saveWithSteps()` 保存处方 + 拆解步骤
- 再创建 queue_item

**重写 callNextTreatment → 废弃**（改为刷卡签到模式）

**新增 checkInByRoom(insuranceCardNo, region)**：
- 按医保卡号查患者 → 查 consultation → 查 prescription_steps WHERE region = currentRegion AND status = 'pending'
- 创建 treatment_record（带 region）
- 返回 { patient, stepsInThisRoom, allSteps, remainingRooms }

**新增 completeRoom(consultationId, region)**：
- 调 `prescriptionStepsService.completeRoomSteps()`
- 调 `checkAllCompleted()`
- 全部完成 → 返回 `{ allDone: true }`
- 未完成 → 返回 `{ allDone: false, pendingRegions, pendingSteps }`

**修改 completeTreatment(patientId)**：
- 保存治疗后生理数据 + 量表结果
- 更新 queue_item → completed
- 更新 patient → completed
- 完成 consultation

if (consultationData.scaleResults) {

    └─ 全部完成 → post-vitals → post-scale → completing → idle
```

#### 步骤 12：联调验证

三个浏览器窗口分别打开治疗终端，各选一个 region：
1. 医生终端开处方（含跨房间项目）→ 确认流转
2. 睡眠区终端刷卡签到 → 看到本房间项目 → 执行 → 完成 → 提示去情志区
3. 情志区终端刷卡签到 → 看到本房间项目 → 执行 → 完成 → 提示去运动区
4. 运动区终端刷卡签到 → 执行 → 完成 → 弹出治疗后采集 → 出队
5. 验证 Realtime：一个终端完成后，其他终端能实时看到 steps 状态更新
