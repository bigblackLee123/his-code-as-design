## 每张表的职责

| 表 | 职责 |
|---|---|
| `patients` | 患者主表，存基本信息 + status 状态机（checked-in → waiting → consulting → pending-treatment → treating → completed） |
| `consultations` | 就诊记录，一次完整流程的聚合锚点。status 只有 `created` / `in-progress` / `completed`。所有业务数据通过 consultation_id 关联 |
| `vital_signs` | 生理数据，通过 `stage` 区分 pre-treatment（分诊采集）和 post-treatment（治疗后采集），1:N 挂在 consultation 下 |
| `contraindications` | 禁忌症字典表，预置数据，支持拼音/首字母搜索 |
| `consultation_contraindications` | 就诊-禁忌症关联表（M:N），医生选择禁忌症时即时写入 |
| `scale_templates` + `scale_questions` | 量表模板 + 题目，预置数据 |
| `scale_results` | 量表填写结果，通过 `stage` 区分 pre（诊前）和 post（治疗后），1:N 挂在 consultation 下 |
| `ai_suggestions` | AI 疗愈建议，1:1 consultation（UNIQUE 约束），存 prompt_snapshot + raw_response 用于审计 |
| `prescriptions` | 处方主表，1:1 consultation（UNIQUE 约束），meta/herbs 为 JSONB |
| `prescription_steps` | 处方执行步骤，1:N 挂在 prescription 下，每个疗愈项目一条，带 region + status 状态机（pending → in-progress → completed/skipped），多房间流转的核心 |
| `therapy_projects` | 疗愈项目库，含 region、mechanism、contraindications JSONB 等 |
| `queue_items` | 统一队列表，通过 `queue_type` 区分 waiting（候诊）和 treatment（治疗），带 region 字段 |
| `treatment_records` | 治疗记录，1:N consultation（每个房间一条），记录 start_time / end_time / duration / region |
| `system_config` | 系统配置表，存 API Key、模型名等可热更换配置 |

## 数据流图一致性对照

逐步对照数据流图 12 步和实际代码实现：

| 步骤 | 数据流图描述 | 实际代码实现 | 一致？ |
|------|------------|------------|--------|
| ① 签到 | `patientService.getByInsuranceCard()` 或 `.create()` → patients 表，status=checked-in | `patientService.ts` 完全一致，create 时自动创建 consultation | ✅ |
| ② 生理数据 | `patientService.saveVitalSigns()` → vital_signs (stage=pre-treatment) | 完全一致，通过 consultationHelper.getActiveId 关联 | ✅ |
| ③ 入候诊队列 | `queueService.enqueueWaiting()` → queue_items (type=waiting)，patient→waiting | `waitingQueueService.ts` 完全一致，含队列上限 20 人检查 | ✅ |
| ④ 叫号接诊 | `queueService.callNextWaiting()` → queue_item→in-progress，patient→consulting | 完全一致 | ✅ |
| ⑤ 禁忌症 | 即时写入 `consultation_contraindications` | `DoctorTerminalPage.handleContraindicationChange` 即时调 `contraindicationService.saveForConsultation()` | ✅ |
| ⑥ 量表评估 | 即时调 `scaleService.saveResult(patientId, result, 'pre')` | `DoctorTerminalPage.handleScaleSubmit` 即时调用 | ✅ |
| ⑦ AI建议→处方→入队 | consultationId 模式调 Edge Function → ai_suggestions 持久化 → `prescriptionService.saveWithSteps()` → `queueService.enqueueTreatment()` | aiService 支持 consultationId 模式；StatusTransition 调 saveWithSteps + enqueueTreatment；Edge Function 自动 upsert ai_suggestions | ✅ |
| ⑦.5 处方拆解 | 自动拆解为 prescription_steps，每个项目一条带 region | `prescriptionStepsService.createSteps()` 完全一致 | ✅ |
| ⑧ 房间签到 | `checkInByRoom(cardSuffix, region)` → 查患者 → 查 steps → 创建 treatment_record → startRoomSteps | `treatmentQueueService.checkInByRoom()` 完全一致，含 patient→treating 状态更新 | ✅ |
| ⑨ 治疗执行 | 本房间 steps pending→in-progress→completed，写 treatment_record | `RoomCompleteCheck` 调 `completeRoom()` → completeRoomSteps + 更新 treatment_record end_time/duration | ✅ |
| ⑩ 流转检查 | `checkAllCompleted()` → 未完成显示提示卡片，全完成进入采集 | `RoomCompleteCheck` 组件完全实现了 partial/done 两种分支 | ✅ |
| ⑪ 治疗后采集 | vital_signs (post-treatment) + scale_results (post) | `QueueComplete` 调 `completeTreatment(patientId, postVitals, postScaleResult)` 写入 DB | ✅ |
| ⑫ 完成出队 | patient→completed，consultation→completed，queue_item→completed | `treatmentQueueService.completeTreatment()` 完全一致 | ✅ |
