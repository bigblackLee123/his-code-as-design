### 任务三：疗愈套餐数据架构重构 + 多房间治疗流转（后端）

数据来源：`曙光医院-音乐疗愈空间内容梳理-2.xlsx`
- 内容表：36 个治疗项目（睡眠区 13 + 情志区 20 + 运动疗愈区 3），含项目级禁忌症
- 套餐表：14 个套餐，每个套餐 2-3 个项目，项目可能跨多个房间

#### 架构讨论结论（3.19）

**核心变化：处方执行从「单次治疗」变为「多房间多步骤流转」**

新增表 `prescription_steps`（处方执行步骤）：
```
prescription_steps
  id (UUID, PK)
  prescription_id (UUID, FK → prescriptions)
  project_id (UUID, FK → therapy_projects)
  region (VARCHAR) - 冗余房间名，方便终端过滤
  sort_order (INTEGER)
  status (ENUM: pending, in-progress, completed, skipped)
  started_at, completed_at (TIMESTAMPTZ)
```

**设计决策：**
1. 同房间多项目 → 合并执行（患者进一次房间做完该房间所有项目）
2. 房间顺序 → 灵活（患者自行决定先去哪个房间）
3. 治疗终端 → 每个房间一台，登录时绑定 region
4. 患者状态 → 保持简单（treating），细粒度靠 prescription_steps 追踪
5. 量表 + 生理数据 → 所有项目全部完成后才采集，在最后一个房间的终端就地完成

**治疗终端流程改造：**
```
刷卡签到（非叫号）
  → 查询本房间待做项目
  → 护士执行治疗
  → 点「本房间完成」→ 批量更新该房间 steps 为 completed
  → 系统检查全部 steps：
     ├─ 还有未完成 → 提示「请前往 XX 区，待做：YY 项目」
     └─ 全部完成 → 弹出治疗后量表 + 生理数据采集 → 出队
```

**禁忌症改造：**
- 从全局字典表改为关联到 therapy_project 级别
- 新增通用禁忌症（全局）：听力损失、耳鸣、急性精神病状态、严重心律失常等
- 新增睡眠区通用禁忌症：身体疼痛/疾病、创伤患者

**待更新的现有表：**
- `therapy_projects`：补全 36 个项目数据，新增项目级禁忌症字段
- `therapy_packages`：重建 14 个套餐
- `therapy_package_items`：新增 matched_symptoms 字段（项目在套餐中的匹配症状）
- `treatment_records`：从 1:1 consultation 改为 1:N（每个 step 可选关联）
- `queue_items`：治疗队列概念弱化，改为「待治疗患者列表」按 region 过滤

### 任务四：百炼平台 AI 推荐对接（前后端联调）

打通「患者数据 → AI 分析 → 推荐疗愈套餐 → 前端展示」链路。

#### 已确认方案

**API 规格：**
- 百炼 Responses API（非 chat completions），兼容 OpenAI 协议
- baseURL: `https://dashscope.aliyuncs.com/api/v2/apps/protocols/compatible-mode/v1`
- 调用方式：`openai.responses.create({ model, input })`
- 响应取值：`response.output_text` 或 `output[0].content[0].text`
- 模型：qwen3.5-plus（均衡性价比）
- 参考文档：`doc/official_API_Doc/aliyun/`

**API Key 管理：**
- 新增 `system_config` 表存储 DASHSCOPE_API_KEY、AI_MODEL_NAME 等配置
- 原则：演示阶段用个人账号，后续切换只改表数据，不改代码
- Edge Function 每次调用时从 system_config 动态获取

**调用链路：**
```
前端「获取AI建议」
  → Supabase Edge Function (consultationId)
  → 查 DB: system_config + vital_signs + contraindications + scale_results + therapy_projects
  → 组装 prompt（患者数据 + 36个疗愈项目描述作为候选）
  → 百炼 Responses API
  → 解析返回（推荐套餐 + 理由）
  → 返回前端
```

**分阶段实施：**
- 第一阶段：疗愈项目描述直接塞 system prompt，不接知识库
- 第二阶段：中医辨证论治知识上传百炼知识库，做 RAG 增强（知识库 API 参考 `aliyun_知识库_API.md`）

**待定：**
- prompt 模板具体设计（输入格式、输出 JSON schema）
- 流式返回 vs 一次性返回
- 前端 AI 推荐卡片的交互细节
