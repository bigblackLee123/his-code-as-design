# 状态管理架构

> 本项目的状态机不依赖数据库引擎，而是通过「DB 枚举约束 + Service 层流转逻辑 + 前端页面状态机」三层协作实现。

---

## 一、架构总览

```
┌─────────────────────────────────────────────────────┐
│  前端页面层（React State）                            │
│  控制 UI 渲染、用户交互、调用时机                       │
│  例：TreatmentTerminalPage.pageStatus                │
├─────────────────────────────────────────────────────┤
│  Service 层（src/services/supabase/）                │
│  控制状态流转顺序、业务规则校验、DB 读写                 │
│  例：waitingQueueService.enqueueWaiting()            │
├─────────────────────────────────────────────────────┤
│  DB 层（PostgreSQL ENUM）                            │
│  约束合法状态值，拒绝非法写入                           │
│  例：patient_status ENUM('checked-in', ...)          │
└─────────────────────────────────────────────────────┘
```

**职责分工：**
- DB 层：管「能存什么值」（ENUM 约束）
- Service 层：管「按什么顺序改值」（流转逻辑）
- 前端页面层：管「什么时候调哪个 Service」（交互驱动）

---

## 二、DB 层：ENUM 类型约束

PostgreSQL ENUM 保证字段值只能是预定义集合之一，写入非法值直接报错。但 ENUM 不管流转顺序。

```sql
-- 患者状态（6 个合法值）
CREATE TYPE patient_status AS ENUM (
  'checked-in','waiting','consulting',
  'pending-treatment','treating','completed'
);

-- 队列状态（3 个合法值）
CREATE TYPE queue_status AS ENUM ('waiting','in-progress','completed');

-- 处方步骤状态（4 个合法值）
CREATE TYPE prescription_step_status AS ENUM (
  'pending','in-progress','completed','skipped'
);

-- 生理数据阶段
CREATE TYPE vital_stage AS ENUM ('pre-treatment','post-treatment');

-- 量表阶段
CREATE TYPE scale_stage AS ENUM ('pre','post');
```

> ⚠️ 当前未添加 DB trigger 校验流转顺序。如需防御性保障，可加 trigger（见第五节）。

## 三、Service 层：状态流转控制

每个 Service 方法只做一步状态变更，前端按业务流程依次调用。

### 3.1 Patient.status 流转

```
checked-in → waiting → consulting → pending-treatment → treating → completed
```

| 流转 | 触发方法 | 所在文件 |
|------|---------|---------|
| → checked-in | `patientService.create()` | `patientService.ts` |
| checked-in → waiting | `waitingQueueService.enqueueWaiting()` | `waitingQueueService.ts` |
| waiting → consulting | `waitingQueueService.callNextWaiting()` | `waitingQueueService.ts` |
| consulting → pending-treatment | `treatmentQueueService.enqueueTreatment()` | `treatmentQueueService.ts` |
| pending-treatment → treating | `treatmentQueueService.checkInByRoom()` | `treatmentQueueService.ts` |
| treating → completed | `treatmentQueueService.completeTreatment()` | `treatmentQueueService.ts` |

### 3.2 QueueItem.status 流转

**候诊队列（queue_type = 'waiting'）：**

| 流转 | 触发方法 |
|------|---------|
| → waiting | `waitingQueueService.enqueueWaiting()` |
| waiting → in-progress | `waitingQueueService.callNextWaiting()` |

**治疗队列（queue_type = 'treatment'）：**

| 流转 | 触发方法 |
|------|---------|
| → waiting | `treatmentQueueService.enqueueTreatment()` |
| waiting → completed | `treatmentQueueService.completeTreatment()` |

### 3.3 PrescriptionStep.status 流转

```
pending → in-progress → completed
                     → skipped
```

| 流转 | 触发方法 | 所在文件 |
|------|---------|---------|
| → pending | `prescriptionStepsService.createSteps()` | `prescriptionStepsService.ts` |
| pending → in-progress | `prescriptionStepsService.startRoomSteps()` | `prescriptionStepsService.ts` |
| in-progress → completed | `prescriptionStepsService.completeRoomSteps()` | `prescriptionStepsService.ts` |

### 3.4 Consultation.status 流转

```
in-progress → completed
```

| 流转 | 触发方法 | 所在文件 |
|------|---------|---------|
| → in-progress | `consultationHelper.create()` | `consultationHelper.ts` |
| in-progress → completed | `consultationHelper.complete()` | `consultationHelper.ts` |

> 注意：consultation 创建即为 `in-progress`，无 `created` 中间态。`complete()` 由 `treatmentQueueService.completeTreatment()` 在出队时调用。

## 四、前端页面层：UI 状态机

前端页面用 React state 维护一个独立的 UI 状态机，驱动 Block 组件的渲染切换。UI 状态机与 DB 状态是解耦的——UI 状态控制「显示什么界面」，DB 状态控制「数据处于什么阶段」。

### 4.1 治疗终端状态机（TreatmentTerminalPage）

```
region-select → idle → checked-in → treating → room-completing
                 ↑                                    │
                 │         ┌──────────────────────────┤
                 │         │ 未全部完成                  │ 全部完成
                 │         ▼                           ▼
                 ├── (等待下一位)              post-vitals → post-scale → completing
                 │                                                          │
                 └──────────────────────────────────────────────────────────┘
```

| UI 状态 | 渲染组件 | 用户操作 | 调用的 Service |
|---------|---------|---------|---------------|
| region-select | RegionSelector | 选择房间 | 无（纯前端） |
| idle | TreatmentQueue | 输入卡号后四位签到 | `treatmentQueueService.checkInByRoom()` |
| checked-in | TreatmentPatientView + TreatmentAction | 点击「开始治疗」 | 无（纯前端计时） |
| treating | TreatmentAction（计时中） | 点击「结束治疗」 | 无（纯前端） |
| room-completing | RoomCompleteCheck | 自动触发 | `treatmentQueueService.completeRoom()` |
| post-vitals | PostVitalSigns | 录入治疗后生理数据 | 无（暂存前端 state） |
| post-scale | PostScaleForm | 填写治疗后量表 | 无（暂存前端 state） |
| completing | QueueComplete | 点击「确认出队」 | `treatmentQueueService.completeTreatment()` |

### 4.2 医生终端状态机（DoctorTerminalPage）

医生终端没有显式的 `pageStatus` 状态机，而是通过 `currentPatient` 是否为 null + `showTransition` 布尔值控制 UI 流程：

```
无患者（currentPatient = null）
  │ 叫号 → callNextWaiting()
  ▼
有患者（currentPatient 存在）
  │ 录入禁忌症 → 即时写 DB（contraindicationService.saveForConsultation）
  │ 提交量表   → 即时写 DB（scaleService.saveResult）
  │ 获取 AI 建议 → Edge Function
  │ 选择疗愈项目
  │ 点击「确认处方并流转」→ showTransition = true
  ▼
StatusTransition 组件
  │ prescriptionService.saveWithSteps()
  │ queueService.enqueueTreatment()
  │ 成功 → onComplete → 重置所有 state
  ▼
回到无患者状态
```

### 4.3 分诊终端状态机（TriageTerminalPage）

分诊终端是线性流程，通过步骤 state 控制：

```
签到（PatientCheckIn）→ 生理数据（VitalSignsInput）→ 入队（QueueAssignment）→ 重置
```

## 五、为什么不在 DB 层强制流转顺序

当前设计选择在 Service 层控制流转，原因：

1. **操作者可信**：内部 HIS 系统，使用者是护士和医生，不存在恶意跳步
2. **业务耦合紧密**：流转条件涉及跨表查询（如「全部 steps completed 才能进 post-vitals」），放在 trigger 里复杂度高
3. **调试友好**：TypeScript 代码比 PL/pgSQL trigger 更容易 debug 和修改
4. **迭代灵活**：业务流程还在快速迭代，Service 层改起来成本低

**如果后续需要 DB 层防御性校验**，可以加 trigger：

```sql
CREATE OR REPLACE FUNCTION check_patient_status_transition()
RETURNS TRIGGER AS $$
DECLARE
  valid_transitions TEXT[][] := ARRAY[
    ['checked-in', 'waiting'],
    ['waiting', 'consulting'],
    ['consulting', 'pending-treatment'],
    ['pending-treatment', 'treating'],
    ['treating', 'completed']
  ];
  i INTEGER;
BEGIN
  FOR i IN 1..array_length(valid_transitions, 1) LOOP
    IF OLD.status::TEXT = valid_transitions[i][1]
       AND NEW.status::TEXT = valid_transitions[i][2] THEN
      RETURN NEW;
    END IF;
  END LOOP;
  RAISE EXCEPTION '非法状态流转: % → %', OLD.status, NEW.status;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER enforce_patient_status
  BEFORE UPDATE OF status ON patients
  FOR EACH ROW EXECUTE FUNCTION check_patient_status_transition();
```

## 六、Realtime 订阅与状态同步

状态变更后，其他终端通过 Supabase Realtime 感知变化：

| 表 | 订阅 Hook | 用途 |
|----|----------|------|
| `queue_items` + `patients` | `useQueueRealtime.ts` | 候诊/治疗队列实时刷新 |
| `prescription_steps` | `usePrescriptionStepsRealtime.ts` | 多房间流转进度同步 |

Realtime 依赖 `REPLICA IDENTITY FULL`（migration 已配置），确保 UPDATE/DELETE 事件包含完整行数据。
