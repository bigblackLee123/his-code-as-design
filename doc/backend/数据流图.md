# HIS 数据流图

> 基于前端三终端交互流程梳理的完整数据流，用于指导 Supabase 后端表结构设计。

---

## 一、患者就诊全流程概览

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  分诊终端  │ ──▶ │  医生终端  │ ──▶ │  治疗终端  │
│  Triage   │     │  Doctor   │     │ Treatment │
└──────────┘     └──────────┘     └──────────┘
  ① 签到          ④ 叫号接诊       ⑧ 叫号治疗
  ② 采集生理数据    ⑤ 禁忌症录入     ⑨ 开始计时
  ③ 入候诊队列     ⑥ 量表评估       ⑩ 治疗后生理数据
                  ⑦ AI建议→开处方   ⑪ 治疗后量表
                     → 入治疗队列    ⑫ 完成出队
```

---

## 二、分诊终端数据流（Triage Terminal）

### 步骤 1：患者签到（PatientCheckIn）

```
输入: 医保卡号 (insuranceCardNo)
  │
  ▼
┌─────────────────────────┐
│ patientService           │
│ .getByInsuranceCard()    │
│ 或 .create()（手动建档）   │
└─────────────────────────┘
  │
  ▼
输出: Patient {
  id, name, gender, age,
  idNumber, phone, insuranceCardNo,
  status: "checked-in",
  createdAt
}
```

写入表: `patients`
状态变更: → `checked-in`

目前前端 PatientCheckIn 组件只有手动输入医保卡号的逻辑，需要补充读卡器硬件接入的完整数据流：

```
┌──────────┐    USB/串口     ┌──────────┐    HTTP/WS     ┌──────────┐
│ 医保卡读卡器 │ ──────────▶ │ 本地驱动/中间件│ ──────────▶ │  前端浏览器  │
│ (硬件)     │              │ (读卡服务)   │              │           │
└──────────┘              └──────────┘              └──────────┘
                                                        │
                                                        ▼
                                                  ┌──────────┐
                                                  │ Supabase  │
                                                  │ 后端查询/建档│
                                                  └──────────┘
```

需要确认的问题：
1. 读卡器品牌/型号？通信协议是 USB HID、串口还是蓝牙？
2. 是否需要本地中间件（如 ActiveX、WebSocket 本地服务）来桥接硬件与浏览器？
3. 读卡器返回的数据格式？（卡号、姓名、身份证号、参保类型等字段）
4. 医保卡数据是否需要调用医保局接口做实时验证？
5. 读卡失败时的降级方案（手动输入 → 后端查询）

### 步骤 2：生理数据采集（VitalSignsInput）

```
输入: Patient + 护士手动录入
  │
  ▼
┌─────────────────────────┐
│ patientService           │
│ .saveVitalSigns()        │
└─────────────────────────┘
  │
  ▼
输出: VitalSigns {
  systolicBP,    // 收缩压 40-300 mmHg
  diastolicBP,   // 舒张压 20-200 mmHg
  heartRate,     // 心率 20-300 次/分
  recordedAt,
  recordedBy
}
```

写入表: `vital_signs`
校验规则: 见 `VITAL_SIGNS_RULES` + `VITAL_SIGNS_ALERT_THRESHOLDS`

### 步骤 3：候诊队列分配（QueueAssignment）

```
输入: Patient + VitalSigns + 科室ID
  │
  ▼
┌─────────────────────────┐
│ queueService             │
│ .enqueueWaiting()        │
└─────────────────────────┘
  │
  ▼
输出: QueueItem {
  id, patientId, patientName,
  insuranceCardNo, queueNumber,
  status: "waiting",
  enqueuedAt
}
```

写入表: `queue_items`
状态变更: Patient → `waiting`
约束: 队列上限 20 人

## 三、医生终端数据流（Doctor Terminal）

### 步骤 4：叫号接诊（CallQueue）

```
输入: 科室ID
  │
  ▼
┌─────────────────────────┐
│ queueService             │
│ .getWaitingQueue()       │
│ .callNextWaiting()       │
└─────────────────────────┘
  │
  ▼
输出: Patient（当前接诊患者）
```

读取表: `queue_items` WHERE status = 'waiting'
状态变更: QueueItem → `in-progress`, Patient → `consulting`

### 步骤 5：禁忌症录入（ContraindicationInput）

```
输入: 拼音/首字母搜索
  │
  ▼
┌─────────────────────────┐
│ contraindicationService  │
│ .search()                │
└─────────────────────────┘
  │
  ▼
输出: Contraindication[] {
  code, name, pinyin,
  pinyinInitial, category
}
```

读取表: `contraindications`（字典表，预置数据）
写入表: `consultation_contraindications`（关联表）

### 步骤 6：量表评估（ScaleForm）

```
输入: 量表模板选择
  │
  ▼
┌─────────────────────────┐
│ scaleService             │
│ .getTemplates()          │
│ .submitResult()          │
└─────────────────────────┘
  │
  ▼
输出: ScaleResult {
  templateId,
  answers: Record<questionId, value>,
  totalScore,
  submittedAt
}
```

读取表: `scale_templates` + `scale_questions`
写入表: `scale_results`

### 步骤 7：AI 建议 → 开处方 → 入治疗队列

```
输入: Patient + ConsultationData {
  contraindications, scaleResults
}
  │
  ▼
┌─────────────────────────┐
│ aiService                │
│ .generateSuggestion()    │
└─────────────────────────┘
  │
  ▼
AISuggestion {
  herbs[], usage, notes,
  confidence, generatedAt
}
  │
  ▼ 医生采纳/修改
┌─────────────────────────┐
│ prescriptionService      │
│ .save()                  │
└─────────────────────────┘
  │
  ▼
PrescriptionData {
  meta: { route, usage, frequency,
          dosage, orderType,
          department, doses },
  herbs: HerbItem[],
  totalAmount
}
  │
  ▼ 处方上传 → StatusTransition
┌─────────────────────────┐
│ queueService             │
│ .enqueueTreatment()      │
└─────────────────────────┘
```

写入表: `ai_suggestions`, `prescriptions`, `prescription_herbs`, `queue_items`
状态变更: Patient → `pending-treatment`

#### ⚠️ 待定：AI 处方建议接入方案

目前 `aiService.getSuggestion()` 是纯 mock 返回硬编码数据，需要接入阿里云百炼（DashScope）真实 API。

可用模型（中国内地部署，base_url: `https://dashscope.aliyuncs.com/compatible-mode/v1`）：

| 模型 | 适用场景 | 输入价格/百万Token | 输出价格/百万Token | 上下文长度 |
|------|---------|-------------------|-------------------|-----------|
| Qwen3-Max | 复杂任务，能力最强 | $0.359 | $1.434 | 262,144 |
| Qwen3.5-Plus | 效果/速度/成本均衡 | $0.115 | $0.688 | 1,000,000 |
| Qwen3.5-Flash | 简单任务，速度快成本低 | $0.029 | $0.287 | 1,000,000 |

API 兼容 OpenAI 协议，Node.js 调用示例：
```js
import OpenAI from "openai";
const client = new OpenAI({
  apiKey: process.env.DASHSCOPE_API_KEY,
  baseURL: "https://dashscope.aliyuncs.com/compatible-mode/v1",
});
const completion = await client.chat.completions.create({
  model: "qwen3.5-plus",  // 推荐：均衡性价比
  messages: [
    { role: "system", content: "..." },
    { role: "user", content: "..." },
  ],
});
```

需要走测试 SOP 确认的三个环节：

**环节 A：API 调用层**
- 调用方式：前端直连 vs Supabase Edge Function 代理？（推荐后者，避免 API Key 暴露）
- 模型选择：Qwen3.5-Plus（均衡）还是 Qwen3.5-Flash（快+便宜）？
- 超时/重试策略？流式返回还是一次性返回？
- API Key 管理：存 Supabase Vault 还是环境变量？

**环节 B：Prompt 工程与数据输入架构**

AI 的输入有两个来源：

来源 1：云端知识库（百炼平台托管）
- 中医辨证论治知识、经典方剂、药物配伍禁忌等
- 直接在百炼平台配置 RAG 知识库，模型调用时自动检索
- 优势：知识更新不需要改代码，平台侧管理即可

来源 2：患者实时数据（Edge Function 组装）
- 由 Supabase Edge Function 从数据库查询后，套入 Prompt 模板发给大模型
- 大模型本身不访问数据库，只接收结构化文本

推荐方案：Edge Function 组装模板（方案 A）

流程：
前端点击"获取AI建议"
  → 调用 Edge Function（传入 consultationId）
  → Edge Function 从 DB 查询：
      - vital_signs（生理数据）
      - consultation_contraindications（禁忌症）
      - scale_results（量表评估）
  → 套入 Prompt 模板：
      """
      ## 患者信息
      - 收缩压: {systolicBP} mmHg
      - 舒张压: {diastolicBP} mmHg
      - 心率: {heartRate} 次/分

      ## 禁忌症
      {contraindications.map(c => c.name).join('、')}

      ## 量表评估
      - 量表: {scaleTemplate.name}
      - 总分: {totalScore}
      - 各项: {answers}

      ## 要求
      请根据以上信息，结合中医辨证论治，给出处方建议。
      返回 JSON 格式：{ herbs: [...], usage, notes, confidence }
      """
  → 调用百炼 API（带知识库 ID）
  → 解析返回 JSON → 返回前端

为什么不让大模型直连数据库（方案 B 排除）：
- 安全风险：大模型有幻觉，可能构造错误查询
- 不可控：无法保证只读取当前患者数据
- 性能：多一轮交互延迟
- 审计：无法精确记录每次传给 AI 的数据快照


**环节 C：前端接收与展示**
- 返回数据解析：JSON.parse 后映射到 `AISuggestion` 类型
- 错误处理：API 超时、返回格式异常、内容安全过滤
- 加载状态：流式逐字显示 vs 骨架屏等待？
- 医生编辑流程：AI 建议 → 医生审核/修改 → 确认采纳 → 填入处方

## 四、治疗终端数据流（Treatment Terminal）

### 步骤 8：叫号治疗（TreatmentQueue）

```
输入: 无（自动拉取治疗队列）
  │
  ▼
┌─────────────────────────┐
│ queueService             │
│ .getTreatmentQueue()     │
│ .callNextTreatment()     │
└─────────────────────────┘
  │
  ▼
输出: TreatmentPatient extends Patient {
  vitalSigns,          // 分诊时采集的
  contraindications,   // 医生录入的
  prescription         // 医生开的处方
}
```

读取表: `queue_items`, `patients`, `vital_signs`, `consultation_contraindications`, `prescriptions`
状态变更: QueueItem → `in-progress`, Patient → `treating`

### 步骤 9-10：治疗过程（TreatmentAction）

```
开始治疗 → 计时 → 结束治疗
  │
  ▼
TreatmentState {
  status: "idle" → "treating" → "post-vitals",
  startTime,
  endTime
}
```

#### ⚠️ 待定：治疗过程中是否加入引导词展示

字段统计中有完整的音乐疗愈引导语数据（引导语原文、速度、情绪、能量/唤醒度等），目前治疗终端的 TreatmentAction 只有计时功能，没有引导词展示。

需要确认的问题：
1. 治疗过程中是否在前端展示引导词？（治疗师朗读 vs 屏幕显示 vs 音频播放）
2. 引导词展示方式：全文一次性显示？还是按段落/节奏逐步滚动？
3. 引导词与治疗计时的关系：是否需要按时间轴同步？（如第 0-2 分钟显示第一段）
4. 数据来源：从 `therapy_projects` 表读取，关联到当前处方对应的疗愈项目
5. 如果需要展示，前端需要新增 Block：`TreatmentGuidance.tsx`

潜在数据流：
```
处方中的疗愈项目 → therapy_projects.引导语原文
                → therapy_projects.速度(BPM)
                → therapy_projects.情绪
                → therapy_projects.能量/唤醒度
                        │
                        ▼
              TreatmentGuidance 组件
              (治疗中实时展示引导词)
```

### 步骤 10：治疗后生理数据（PostVitalSigns）

```
输入: 治疗后护士录入 + 治疗前数据对比
  │
  ▼
输出: VitalSigns（治疗后）
```

写入表: `vital_signs`（type = 'post-treatment'）

### 步骤 11：治疗后量表（PostScaleForm）

```
输入: 量表模板
  │
  ▼
输出: ScaleResult（治疗后评估）
```

写入表: `scale_results`（type = 'post-treatment'）

### 步骤 12：完成出队（QueueComplete）

```
输入: TreatmentPatient + TreatmentState + PostVitals
  │
  ▼
┌─────────────────────────┐
│ 汇总 TreatmentRecord     │
│ queueService             │
│ .completeTreatment()     │
└─────────────────────────┘
  │
  ▼
TreatmentRecord {
  patientId,
  startTime, endTime, duration,
  preVitals, postVitals,
  postScaleResult
}
```

写入表: `treatment_records`
状态变更: QueueItem → `completed`, Patient → `completed`

## 五、数据实体关系总览

```
patients
  │
  ├──1:N──▶ vital_signs          (分诊前 + 治疗后)
  ├──1:N──▶ queue_items          (候诊队列 + 治疗队列)
  ├──1:N──▶ scale_results        (诊前量表 + 治疗后量表)
  ├──1:N──▶ prescriptions        (处方)
  │            └──1:N──▶ prescription_herbs
  ├──1:N──▶ treatment_records    (治疗记录)
  └──1:N──▶ consultations        (就诊记录，串联一次完整流程)
               ├──M:N──▶ contraindications (通过关联表)
               └──1:1──▶ ai_suggestions
```

## 六、核心表清单与字段映射

## 七、状态机

### Patient.status 流转

```
checked-in ──▶ waiting ──▶ consulting ──▶ pending-treatment ──▶ treating ──▶ completed
   分诊签到     入候诊队列    医生叫号        处方上传/入治疗队列    治疗叫号      治疗完成
```

### QueueItem.status 流转

```
候诊队列: waiting ──▶ in-progress ──▶ (出队，患者转入治疗队列)
治疗队列: waiting ──▶ in-progress ──▶ completed
```

## 八、音乐疗愈扩展数据（字段统计补充）

以下数据来自字段统计文档中的音乐疗愈部分，后续需要额外建表：

| 表名 | 说明 |
|------|------|
| `therapy_projects` | 疗愈项目（区域/名称/机制/引导语/速度/情绪/能量等） |
| `therapy_packages` | 疗愈套餐（套餐名称/适用人群/匹配症状） |
| `therapy_package_items` | 套餐-项目关联表 |

这部分与主诊疗流程独立，可在治疗终端中作为处方执行的具体内容扩展。
