# HIS 数据流图

> 基于前端三终端交互流程梳理的完整数据流，用于指导 Supabase 后端表结构设计。

---

## 一、患者就诊全流程概览

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  分诊终端  │ ──▶ │  医生终端  │ ──▶ │  治疗终端  │
│  Triage   │     │  Doctor   │     │ Treatment │
└──────────┘     └──────────┘     └──────────┘
  ① 签到          ④ 叫号接诊       ⑧ 房间签到
  ② 采集生理数据    ⑤ 禁忌症录入     ⑨ 开始计时
  ③ 入候诊队列     ⑥ 量表评估       ⑩ 治疗后生理数据
                  ⑦ AI建议→开处方   ⑪ 治疗后量表
                     → 入治疗队列    ⑫ 完成出队
```

## 二、分诊终端数据流（Triage Terminal）

### 步骤 1：患者签到（PatientCheckIn）

```
输入: 医保卡号 (insuranceCardNo)
  │
  ▼
┌─────────────────────────┐
│ patientService           │
│ .getByInsuranceCard()    │
│ 或 .create()（手动建档）   │
└─────────────────────────┘
  │
  ▼
输出: Patient {
  id, name, gender, age,
  idNumber, phone, insuranceCardNo,
  status: "checked-in",
  createdAt
}
```

写入表: `patients`
状态变更: → `checked-in`

### 步骤 2：生理数据采集（VitalSignsInput）

```
输入: Patient + 护士手动录入
  │
  ▼
┌─────────────────────────┐
│ patientService           │
│ .saveVitalSigns()        │
└─────────────────────────┘
  │
  ▼
输出: VitalSigns {
  systolicBP,    // 收缩压 40-300 mmHg
  diastolicBP,   // 舒张压 20-200 mmHg
  heartRate,     // 心率 20-300 次/分
  recordedAt,
  recordedBy
}
```

写入表: `vital_signs`
校验规则: 见 `VITAL_SIGNS_RULES` + `VITAL_SIGNS_ALERT_THRESHOLDS`

### 步骤 3：候诊队列分配（QueueAssignment）

```
输入: Patient + VitalSigns + 科室ID
  │
  ▼
┌─────────────────────────┐
│ queueService             │
│ .enqueueWaiting()        │
└─────────────────────────┘
  │
  ▼
输出: QueueItem {
  id, patientId, patientName,
  insuranceCardNo, queueNumber,
  status: "waiting",
  enqueuedAt
}
```

写入表: `queue_items`
状态变更: Patient → `waiting`
约束: 队列上限 20 人

## 三、医生终端数据流（Doctor Terminal）

### 步骤 4：叫号接诊（CallQueue）

```
输入: 科室ID
  │
  ▼
┌─────────────────────────┐
│ queueService             │
│ .getWaitingQueue()       │
│ .callNextWaiting()       │
└─────────────────────────┘
  │
  ▼
输出: Patient（当前接诊患者）
```

读取表: `queue_items` WHERE status = 'waiting'
状态变更: QueueItem → `in-progress`, Patient → `consulting`

### 步骤 5：禁忌症录入（ContraindicationInput）

```
输入: 拼音/首字母搜索
  │
  ▼
┌─────────────────────────┐
│ contraindicationService  │
│ .search()                │
└─────────────────────────┘
  │
  ▼
输出: Contraindication[] {
  code, name, pinyin,
  pinyinInitial, category
}
```

读取表: `contraindications`（字典表，预置数据）
写入表: `consultation_contraindications`（关联表）

### 步骤 6：量表评估（ScaleForm）

```
输入: 量表模板选择
  │
  ▼
┌─────────────────────────┐
│ scaleService             │
│ .getTemplates()          │
│ .submitResult()          │
└─────────────────────────┘
  │
  ▼
输出: ScaleResult {
  templateId,
  answers: Record<questionId, value>,
  totalScore,
  submittedAt
}
```

读取表: `scale_templates` + `scale_questions`
写入表: `scale_results`

### 步骤 7：AI 建议 → 开处方 → 入治疗队列

```
输入: Patient + ConsultationData {
  contraindications, scaleResults
}
  │
  ▼
┌─────────────────────────┐
│ aiService                │
│ .generateSuggestion()    │
└─────────────────────────┘
  │
  ▼
AISuggestion {
  herbs[], usage, notes,
  confidence, generatedAt
}
  │
  ▼ 医生采纳/修改
┌─────────────────────────┐
│ prescriptionService      │
│ .save()                  │
└─────────────────────────┘
  │
  ▼
PrescriptionData {
  meta: { route, usage, frequency,
          dosage, orderType,
          department, doses },
  herbs: HerbItem[],
  totalAmount
}
  │
  ▼ 处方上传 → StatusTransition
┌─────────────────────────┐
│ queueService             │
│ .enqueueTreatment()      │
└─────────────────────────┘
```

写入表: `ai_suggestions`, `prescriptions`, `queue_items`
状态变更: Patient → `pending-treatment`

#### AI 处方建议接入方案（百炼平台）

已确认使用阿里云百炼（DashScope）Responses API，兼容 OpenAI 协议。
API 参考文档：`doc/official_API_Doc/aliyun/`

**API 调用方式：**

```js
import OpenAI from "openai";
const openai = new OpenAI({
  apiKey: apiKeyFromDB,  // 从 system_config 表动态获取
  baseURL: "https://dashscope.aliyuncs.com/api/v2/apps/protocols/compatible-mode/v1"
});
const response = await openai.responses.create({
  model: "qwen3.5-plus",
  input: promptText
});
// 获取回复
const reply = response.output_text;
```

```json
{
  "id": "c9f9c06b-...",
  "model": "qwen3.5-plus",
  "status": "completed",
  "output": [{
    "content": [{ "text": "...", "type": "output_text" }],
    "role": "assistant",
    "status": "completed"
  }],
  "usage": { "input_tokens": 55, "output_tokens": 43, "total_tokens": 98 }
}
```

注意：百炼新版使用 responses.create + input 参数，而非 chat.completions.create + messages。 回复内容通过 response.output_text 获取（SDK 封装），或从 output[0].content[0].text 手动提取。

API Key 管理：

新增 system_config 表存储可热更换的配置：
system_config
  id (UUID, PK)
  key (VARCHAR, UNIQUE) - 如 'DASHSCOPE_API_KEY', 'AI_MODEL_NAME'
  value (TEXT) - 加密存储敏感值
  description (TEXT)
  updated_at (TIMESTAMPTZ)

调用架构：Supabase Edge Function 代理

前端点击"获取AI建议"
  → 调用 Edge Function（传入 consultationId）
  → Edge Function 从 DB 查询：
      - system_config（获取 API Key + 模型名）
      - vital_signs（生理数据）
      - consultation_contraindications（禁忌症）
      - scale_results（量表评估）
      - therapy_projects（全部疗愈项目描述，作为推荐候选）
  → 组装 prompt（患者数据 + 疗愈项目库）
  → 调用百炼 Responses API
  → 解析返回 → 返回前端


百炼平台支持 RAG 知识库，通过 @alicloud/bailian20231229 SDK 管理：

上传文件 → 解析 → 建索引 → 模型调用时自动检索
需要 AccessKey（非 DashScope API Key），适合一次性操作
第一阶段先不接知识库，36 个疗愈项目直接塞 system prompt
第二阶段将中医辨证论治知识上传知识库做 RAG 增强
API 参考：doc/official_API_Doc/aliyun/aliyun_知识库_API.md

**环节 C：前端接收与展示**
- 返回数据解析：JSON.parse 后映射到 `AISuggestion` 类型
- 错误处理：API 超时、返回格式异常、内容安全过滤
- 加载状态：流式逐字显示 vs 骨架屏等待？
- 医生编辑流程：AI 建议 → 医生审核/修改 → 确认采纳 → 填入处方

## 四、治疗终端数据流（Treatment Terminal）

> 重大变更：治疗终端从「统一叫号」改为「多房间刷卡签到」模式。
> 每个房间（睡眠室/情志区/运动疗愈区）独立部署一台终端，登录时绑定 region。
> 一个处方可能包含跨多个房间的项目，患者自行决定房间顺序。

### 步骤 7.5：处方拆解为执行步骤（自动）

```
医生确认处方（含疗愈项目列表）
  │
  ▼
┌─────────────────────────┐
│ prescriptionService      │
│ .createWithSteps()       │
└─────────────────────────┘
  │
  ▼
自动拆解为 prescription_steps
输出: PrescriptionStep[] {
  {project:"舒曼波", region:"睡眠区", status:"pending", sort:1},
  {project:"静心纯音乐", region:"情志区", status:"pending", sort:2},
  {project:"A1", region:"情志区", status:"pending", sort:3},
  {project:"步态调整", region:"运动疗愈区", status:"pending", sort:4},
}
```

写入表: `prescriptions`, `prescription_steps`
状态变更: Patient → `pending-treatment`

### 步骤 8：房间签到（刷卡模式，替代原叫号）

```
患者到达某房间，刷医保卡
  │
  ▼
┌─────────────────────────┐
│ treatmentService         │
│ .checkInByRoom(          │
│   insuranceCardNo,       │
│   currentRegion          │
│ )                        │
└─────────────────────────┘
  │
  ▼
输出: RoomCheckIn {
  patient,
  stepsInThisRoom: PrescriptionStep[],  // 本房间待做项目
  allSteps: PrescriptionStep[],         // 全部项目（含进度）
  remainingRooms: string[]              // 还需去的房间
}
```

读取表: `patients`, `prescription_steps` WHERE region = currentRegion AND status = 'pending'
显示: 护士看到该患者在本房间需要做的所有项目

### 步骤 9：执行治疗（本房间所有项目合并执行）

```
护士确认开始治疗
  │
  ▼
本房间所有 pending steps → status: 'in-progress'
  │
  ▼
治疗过程（计时 + 可选引导词展示）
  │
  ▼
护士点击「本房间完成」
  │
  ▼
┌─────────────────────────┐
│ treatmentService         │
│ .completeRoomSteps(      │
│   consultationId,        │
│   currentRegion          │
│ )                        │
└─────────────────────────┘
  │
  ▼
本房间所有 steps → status: 'completed'
写入: treatment_records（本房间的治疗记录）
```

写入表: `prescription_steps`（批量更新）, `treatment_records`
注意: treatment_records 从 1:1 consultation 改为 1:N（每个房间一条记录）

### 步骤 10：流转检查（自动）

```
本房间完成后，系统自动检查
  │
  ▼
┌─────────────────────────┐
│ treatmentService         │
│ .checkAllStepsCompleted( │
│   consultationId         │
│ )                        │
└─────────────────────────┘
  │
  ├── 还有 pending steps
  │   → 终端显示提示卡片：
  │   「请前往 [情志区]，待做项目：静心纯音乐、A1」
  │   「请前往 [运动疗愈区]，待做项目：步态调整」
  │   → 患者离开，去下一个房间
  │
  └── 全部 steps completed
      → 进入步骤 11（治疗后数据采集）
```

### 步骤 11：治疗后数据采集（仅在最后一个房间触发）

```
全部项目完成 → 当前终端自动弹出采集表单
  │
  ├── 治疗后生理数据（PostVitalSigns）
  │   写入: vital_signs (stage = 'post-treatment')
  │
  └── 治疗后量表（PostScaleForm）
      写入: scale_results (stage = 'post')
```

注意: 最后一个房间可能是任意房间（顺序灵活），每个终端都具备采集能力。

### 步骤 12：完成出队

```
治疗后数据采集完成
  │
  ▼
┌─────────────────────────┐
│ treatmentService         │
│ .completeConsultation(   │
│   consultationId         │
│ )                        │
└─────────────────────────┘
  │
  ▼
汇总: 所有 treatment_records + postVitals + postScaleResult
状态变更: Patient → 'completed'
```

## 五、数据实体关系总览

```
patients
  │
  ├──1:N──▶ consultations        (就诊记录，串联一次完整流程)
  │            ├──1:1──▶ ai_suggestions
  │            ├──1:1──▶ prescriptions
  │            │            └──1:N──▶ prescription_steps (多房间流转核心)
  │            ├──M:N──▶ contraindications (通过 consultation_contraindications)
  │            ├──1:N──▶ vital_signs      (分诊前 + 治疗后)
  │            ├──1:N──▶ scale_results    (诊前量表 + 治疗后量表)
  │            └──1:N──▶ treatment_records (每个房间一条)
  │
  └──1:N──▶ queue_items          (候诊队列 + 治疗队列)

独立字典表:
  contraindications (禁忌症字典，预置数据)
  therapy_projects  (疗愈项目库，含 contraindications JSONB)
  scale_templates → scale_questions (量表模板)
  system_config   (系统配置，API Key 等)
```

## 七、状态机

### Patient.status 流转

```
checked-in ──▶ waiting ──▶ consulting ──▶ pending-treatment ──▶ treating ──▶ completed
   分诊签到     入候诊队列    医生叫号        处方上传/入治疗队列    房间签到      治疗完成
```

### QueueItem.status 流转

```
候诊队列: waiting ──▶ in-progress ──▶ (出队，患者转入治疗队列)
治疗队列: waiting ──▶ in-progress ──▶ completed
```

### PrescriptionStep.status 流转

```
pending ──▶ in-progress ──▶ completed
                        ──▶ skipped（跳过）
```
