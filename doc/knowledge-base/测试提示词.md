# AI 疗愈推荐 — 测试提示词

> 用于测试 `ai-therapy-suggestion` Edge Function 的各种患者场景。
> 每个测试用例包含：场景描述、请求 body、预期行为。
>
> **架构说明**：Edge Function 在调用大模型前，会先查 `therapy_projects` 表并用代码排除禁忌症冲突项目，
> 只将合格的候选列表传给大模型。因此禁忌排除是确定性的，不依赖模型判断。

---

## 测试方式

通过 curl 直接调用本地 Edge Function：

```bash
curl -X POST http://127.0.0.1:54321/functions/v1/ai-therapy-suggestion \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <ANON_KEY>" \
  -d '<请求体>'
```

## 场景 1：高血压 + 焦虑（典型镇静需求）

**描述**：收缩压偏高、心率偏快，量表评分较高（焦虑倾向），无禁忌症。应推荐低 BPM 镇静类项目。

```json
{
  "vitals": { "systolicBP": 155, "diastolicBP": 95, "heartRate": 92 },
  "contraindications": [],
  "scaleResult": { "totalScore": 28 }
}
```

**预期**：
- 无禁忌 → 候选列表包含全部 36 个项目
- 推荐项目 BPM ≤ 55，偏向深度镇静/平静放松
- 应覆盖多个区域（如睡眠区 + 情志区）
- reason 中应提及"血压偏高""心率偏快""镇静"等关键词
- confidence ≥ 0.8
- projectIds 为数据库 UUID 格式

---

## 场景 2：低血压 + 精力不足（激活需求）

**描述**：血压偏低、心率偏慢，量表评分中等，无禁忌症。应推荐中度激活类项目。

```json
{
  "vitals": { "systolicBP": 95, "diastolicBP": 58, "heartRate": 56 },
  "contraindications": [],
  "scaleResult": { "totalScore": 15 }
}
```

**预期**：
- 无禁忌 → 候选列表包含全部 36 个项目
- 推荐项目中应包含中高 BPM 项目（方形呼吸 BPM 115、穿越山林 BPM 90 等）
- reason 中应提及"血压偏低""精力不足""激活"等关键词
- 不应全部推荐深度镇静类

---

## 场景 3：有禁忌症 — 创伤后应激障碍（PTSD）

**描述**：生理指标正常，但患者有 PTSD 禁忌。代码层面会排除所有含 PTSD 禁忌的项目。

```json
{
  "vitals": { "systolicBP": 120, "diastolicBP": 78, "heartRate": 72 },
  "contraindications": [{ "name": "创伤后应激障碍" }],
  "scaleResult": { "totalScore": 20 }
}
```

**预期**：
- 代码排除后，候选列表中不会出现任何含"创伤后应激障碍"禁忌的项目
- 剩余可选项目约为：腿部按摩、体感音波、腰部按摩、A1、B1、心率牵引-01、古风《做个神仙》等
- 推荐结果必须全部来自候选列表
- reason 中应说明可选范围受限

## 场景 4：多重禁忌症叠加（耳鸣）

**描述**：患者有耳鸣禁忌。耳鸣是全局禁忌，所有项目都包含此禁忌，代码过滤后候选列表为空。

```json
{
  "vitals": { "systolicBP": 130, "diastolicBP": 82, "heartRate": 68 },
  "contraindications": [{ "name": "耳鸣" }],
  "scaleResult": { "totalScore": 35 }
}
```

**预期**：
- 代码过滤后候选列表为空（0 个项目）
- user message 中显示"无可用项目，所有项目均与患者禁忌症冲突"
- AI 应返回空 projectIds 或在 reason 中说明无法推荐
- confidence 应明显偏低（< 0.5）
- 验证系统不会崩溃，能优雅处理空候选

## 场景 5：数据缺失 — 无生理数据

**描述**：只有量表评分，无 vitals。测试 AI 在部分数据缺失时的容错能力。

```json
{
  "contraindications": [],
  "scaleResult": { "totalScore": 22 }
}
```

**预期**：
- 函数不应报错（vitals 为 null，prompt 中显示"暂无"）
- 候选列表正常（无禁忌 → 全部 36 个）
- AI 应仅基于量表评分推荐，reason 中说明生理数据缺失
- 推荐仍应合理（基于评分偏高，倾向放松类）

## 场景 6：数据缺失 — 无量表评分

**描述**：有完整 vitals 但无量表，有精神分裂症禁忌。

```json
{
  "vitals": { "systolicBP": 140, "diastolicBP": 88, "heartRate": 85 },
  "contraindications": [{ "name": "精神分裂症患者" }]
}
```

**预期**：
- scaleScore 为 null，prompt 中显示"暂无"
- 代码排除精神分裂症禁忌项目（方形呼吸、穿越山林、B1、古风、432Hz、运动疗愈区全部）
- 应基于血压偏高推荐低 BPM 镇静类
- reason 中应提及量表数据缺失

## 场景 7：正常指标 + 无禁忌（最大自由度）

**描述**：生理指标正常，评分中等，无禁忌。期望推荐覆盖多区域。

```json
{
  "vitals": { "systolicBP": 125, "diastolicBP": 76, "heartRate": 70 },
  "contraindications": [],
  "scaleResult": { "totalScore": 18 }
}
```

**预期**：
- 候选列表包含全部 36 个项目
- 理想情况下覆盖三个区域（睡眠区 + 情志区 + 运动疗愈区）
- 验证 AI 是否能做到"多维度干预"的要求
- 注意：运动疗愈区项目适用人群为 50/60 岁以上，观察 AI 是否提及此限制

## 场景 8：通过 consultationId 调用（DB 模式）

**描述**：传入 consultationId，测试从数据库聚合患者上下文 + 查项目过滤的完整链路。

```json
{
  "consultationId": "<替换为实际测试就诊ID>"
}
```

**前置条件**：
- `consultations` 表中存在该记录，关联 `patients` 表
- `vital_signs` 表中有对应 pre-treatment 阶段数据
- `consultation_contraindications` 表中有禁忌症关联
- `scale_results` 表中有 pre 阶段量表评分
- `therapy_projects` 表中有项目数据（含 contraindications JSONB）

**预期**：
- 函数从 DB 正确聚合患者数据
- 禁忌过滤基于 DB 中的 contraindications 字段
- 返回结果写入 `ai_suggestions` 表（upsert）
- 再次调用同一 consultationId 应覆盖而非新增

## 场景 9：JSON 格式验证

**描述**：验证 AI 返回严格符合约定的 JSON 结构。

使用场景 1 的请求体，重点检查返回：

```json
{
  "id": "AI-<timestamp>",
  "projectIds": ["UUID 格式"],
  "projectNames": ["知识库中真实项目名"],
  "reason": "100-200字推荐理由",
  "confidence": 0.85,
  "generatedAt": "ISO 8601"
}
```

**检查项**：
- `projectIds` 和 `projectNames` 长度一致且为 2-4 个
- `projectIds` 为合法 UUID 格式
- `projectNames` 全部存在于 `therapy_projects` 表中
- `reason` 字数在 100-200 字之间
- `confidence` 为 0-1 之间的数值
- `generatedAt` 为合法 ISO 时间戳
- `id` 以 "AI-" 开头
